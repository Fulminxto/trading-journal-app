# VOLTIS Codex Motion Rules

This document defines how Codex should think before implementing or modifying any interaction, animation or transition inside VOLTIS.

The objective is not to create impressive motion.

The objective is to create trust.

Every implementation must reinforce the identity of VOLTIS.

When multiple solutions are technically correct, always choose the one that best reflects the VOLTIS Design System.

## Mindset

Before writing code, remember:

VOLTIS is not a website.

VOLTIS is not a landing page.

VOLTIS is not a portfolio.

VOLTIS is a professional operating system.

Every interaction should increase confidence.

Every animation should support productivity.

Nothing should exist simply because it looks impressive.

Premium is communicated through restraint.

## Before Writing Code

Before implementing any motion:

Understand the user's objective.

Understand the component's role.

Understand what the animation needs to communicate.

Do not start from the animation.

Start from the user experience.

Ask yourself:

- Why should this move?
- What information does motion communicate?
- Would the interface become worse without it?

If motion has no functional purpose, do not add it.

## During Implementation

Always prioritize:

- continuity
- consistency
- predictability
- responsiveness
- accessibility

Preserve orientation.

Never surprise the user.

Never interrupt workflow.

Prefer subtle motion over dramatic motion.

Prefer believable movement over decorative movement.

Prefer refinement over complexity.

Motion should feel engineered.

## Before Finishing

Before considering the task complete, verify:

- The interaction feels natural.
- The animation improves clarity.
- Motion does not reduce usability.
- Keyboard interaction still works.
- Reduced Motion is respected.
- Mobile interaction remains intuitive.
- Focus management is preserved.
- No layout shift was introduced.
- The implementation is consistent with nearby components.

## Forbidden Decisions

Never introduce motion simply because it looks modern.

Never copy interactions from another product without adaptation.

Never sacrifice usability for visual impact.

Never add unnecessary delay.

Never use excessive bounce.

Never overuse scaling.

Never rotate elements without functional purpose.

Never animate every visible element.

Never create visual noise.

Never redesign unrelated components.

Never ignore the existing Design System.

Never ignore the Motion Language.

Never ignore accessibility.

## Final Decision

Before delivering the implementation, ask:

Does this feel like VOLTIS?

Would this interaction still feel premium after one thousand uses?

Would a professional trader appreciate this refinement?

Does this increase trust?

Does this reduce cognitive effort?

Would removing the animation make the experience worse?

If any answer is "No",

the implementation is not finished.

## Self Review Process

Before completing any interface implementation, perform an internal design review.

Review the result as if you were the Lead Product Designer of VOLTIS.

Do not evaluate whether the code works.

Evaluate whether the experience feels correct.

Check the implementation against:

- Motion Language
- Motion Patterns
- Design Dictionary
- Physical Language

If inconsistencies exist, refine the implementation before considering the task complete.

## Design Quality Score

Before considering the task complete, internally evaluate the implementation.

Score each category from 1 to 10.

- Visual Quality
- Motion Quality
- Consistency
- Premium Feeling
- Physical Believability
- Accessibility
- Responsiveness
- Performance
- Craftsmanship

If any category is below 9,

continue refining the implementation.

The objective is not perfection.

The objective is excellence.

## Design Review Questions

Before finishing, ask:

Does this interaction feel engineered?

Does it feel expensive?

Does it feel effortless?

Does it feel calm?

Does it communicate confidence?

Would this interaction strengthen the identity of VOLTIS?

Would this still feel correct after one thousand uses?

Would removing this motion make the experience worse?

Would this interaction deserve to become part of the VOLTIS Design System?

Would the VOLTIS Design Language approve this interaction?

Would I still be proud of this implementation one year from now?
